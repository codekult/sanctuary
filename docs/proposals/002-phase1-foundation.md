# Proposal 002 — Phase 1: Foundation

**Status**: Accepted
**Date**: 2026-03-03
**Depends on**: [001-product-proposal](./001-product-proposal.md)

## Goal

Set up the monorepo, database schema, authentication, API scaffolding, and seed data. At the end of this phase, we have a running dev environment with the full data model, working auth, and a few real taxa imported from iNaturalist — but no UI yet.

## Scope

### Included
- Turborepo monorepo with bun workspaces
- Next.js 15 app scaffolding (App Router)
- Expo/React Native app scaffolding (minimal — just the project shell)
- Drizzle ORM schema for all core entities
- Supabase project setup (Postgres, Auth, Storage)
- tRPC router scaffolding with basic CRUD for each entity
- Zod validation schemas (shared package)
- iNaturalist API integration for taxa search/import
- Seed script that imports sample taxa
- Bilingual support groundwork (i18n setup for EN/ES)
- Dev tooling (ESLint, TypeScript strict, Prettier)

### Excluded
- Any UI (Phase 2)
- Phenology calendar visualization (Phase 4)
- Mobile functionality beyond project shell (Phase 5)
- Image upload flow (Phase 2 — needs UI)
- Deployment to production (Phase 6)

### 2. Database Schema

### 3. tRPC Routers

Each router provides standard CRUD + entity-specific queries:

**taxon router**:
- `taxon.list` — paginated, filterable by kingdom, rank, search text
- `taxon.getById` — single taxon with related individuals count
- `taxon.create` — manual entry
- `taxon.update`
- `taxon.delete`
- `taxon.searchExternal` — search iNaturalist API, return candidates
- `taxon.importFromExternal` — import a taxon from iNaturalist by ID

**individual router**:
- `individual.list` — paginated, filterable by taxon, status
- `individual.getById` — with observation history
- `individual.create`
- `individual.update`
- `individual.delete`

**observation router**:
- `observation.list` — paginated, filterable by taxon, individual, date range, observer
- `observation.getById` — with media, phenology events
- `observation.create`
- `observation.update`
- `observation.delete`

**phenology router**:
- `phenology.listEvents` — filterable by individual, taxon, event type, date range
- `phenology.createEvent`
- `phenology.listEventTypes`
- `phenology.createEventType`
- `phenology.updateEventType`

**media router**:
- `media.list` — by observation or individual
- `media.create` — generate signed upload URL
- `media.delete`
- `media.reorder`

**property router**:
- `property.get` — single property (we only have one)
- `property.update`

All mutation routes require authentication. Read routes are public but **coordinates are stripped from responses for unauthenticated requests** (location privacy decision — see proposal 001).

### 4. iNaturalist API Integration

Use the iNaturalist API v1 (public, no auth required for read):
- `GET https://api.inaturalist.org/v1/taxa/autocomplete?q={query}` — search taxa
- `GET https://api.inaturalist.org/v1/taxa/{id}` — get full taxon details

Map iNaturalist response to our Taxon schema:
- `name` → `scientific_name`
- `preferred_common_name` → `common_name_en`
- `names` array filtered by `locale=es` → `common_name_es`
- `rank` → `taxon_rank`
- `ancestors` → `kingdom`, `phylum`, `class`, `order`, `family`, `genus`
- `id` → `external_id`
- `default_photo.medium_url` → `thumbnail_url`
- `conservation_status` → `conservation_status`
- `wikipedia_summary` → `description_en`

### 5. Auth Setup

- Supabase Auth with email/password
- No public registration — admin invites contributors via Supabase dashboard or a future admin UI
- tRPC middleware: `publicProcedure` (no auth) and `protectedProcedure` (requires valid session)
- User record created in `users` table on first authenticated API call (sync from Supabase Auth)

### 6. i18n Groundwork

- Shared `i18n` package with JSON locale files
- Bilingual text fields in DB (e.g., `common_name_en`, `common_name_es`) rather than a separate translations table — simpler for two languages
- If a third language is ever needed, we'd migrate to a translations table (document this in an ADR)

### 7. Seed Script

The seed script should:
1. Create one property (with sample coordinates and timezone)
2. Import 10-15 diverse taxa from iNaturalist spanning multiple kingdoms (a tree, a bird, a fungus, an insect, a wildflower, etc.)
3. Create 2-3 sample individuals (e.g., a named tree, a nesting bird)
4. Create 5-10 sample observations across different dates
5. Create a few phenology event types (Flowering, Fruiting, Nesting, Dormancy)
6. Create a sample admin user

### 8. Dev Tooling

- TypeScript strict mode across all packages
- ESLint with shared config (no default exports rule, import order)
- Prettier for formatting
- Bun as runtime and package manager (see ADR-007)
- `.env.example` with all required env vars documented
- `README.md` in repo root with setup instructions
